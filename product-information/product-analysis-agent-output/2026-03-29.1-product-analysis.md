# Product Analysis — 2026-03-29 (Analysis #1)

## Executive Summary

The codebase is well-structured and demonstrates a solid foundation: the Angular frontend consistently uses the signals API, dependency injection follows modern `inject()` patterns, services correctly encapsulate HTTP logic, and sensitive configuration is fully externalised via environment variables. The most critical finding is that `PlaidItem.accessToken` is persisted in plain text in the database — a high-severity security concern that needs encryption or a secrets vault. Secondary concerns include pervasive use of `RuntimeException` instead of typed domain exceptions in the backend, `standalone: true` / `CommonModule` still present in multiple shared components, and a near-total absence of automated tests across both frontend and backend.

### 4. PostgreSQL Good Practices

| High | `backend/…/db/changelog/20260124.1-CreatePlaidItemsTable.sql` | `access_token character varying(255)` — Plaid access tokens are sensitive credentials stored in plain text. | Encrypt the column at-rest using pgcrypto or store the token via a secrets manager (e.g., AWS Secrets Manager), never as raw `VARCHAR`. |
| High | `backend/…/db/changelog/20260124.1-CreatePlaidItemsTable.sql` | `plaid_items.user_id` was added as a nullable column (`ADD COLUMN user_id BIGINT`) in migration `20260214.2`. ## Recommended Next Steps

1. **Encrypt the Plaid access token** (`plaid_items.access_token`) — this is the single highest-risk finding. A compromised database currently exposes full API-level access to every user's bank account. Encrypt the column with `pgcrypto` or migrate to a secrets manager before any production deployment.

2. **Add `NOT NULL` + `ON DELETE CASCADE` to `plaid_items.user_id`** — the nullable FK and absent cascade rule will result in orphaned `plaid_items` rows when users are deleted, and silently allows rows to be saved with no owner. Add a follow-up migration: backfill, set `NOT NULL`, and add `ON DELETE CASCADE`.

3. **Replace `RuntimeException` with typed domain exceptions in `PlaidService`** — all `throw new RuntimeException(...)` calls should become `DataSourceException` (which already exists) so the controller and future callers can differentiate Plaid API failures from programmer errors and return appropriate HTTP status codes.

4. **Write tests for the auth and Plaid critical paths** — `AuthController`, `UserService`, `PlaidController`, `PlaidService`, and `AuthGuard` have zero test coverage between them and represent nearly the entire security-sensitive and revenue-critical surface of the application. These should be the first items in a test sprint.

5. **Remove `standalone: true` and `CommonModule` from all four shared components** (`modal`, `input`, `select`, `button`) — these are used everywhere in the app and set a bad precedent. Cleaning them up brings the entire codebase into compliance with the Angular 20+ project conventions.
