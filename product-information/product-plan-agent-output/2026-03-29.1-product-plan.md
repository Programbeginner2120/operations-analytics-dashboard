# Remediation Plan — 2026-03-29 (Plan #1)

> Source analysis: [product-information/product-analysis-agent-output/2026-03-29.1-product-analysis.md](../product-analysis-agent-output/2026-03-29.1-product-analysis.md)

## Executive Summary

The analysis identified **59 raw findings** (de-duplicated to ~47 unique issues) across security, schema integrity, backend correctness, API consistency, Angular compliance, frontend hygiene, and test coverage. The single highest-risk finding — a Plaid access token stored as plain text in the database — must be addressed before any production deployment, as a compromised database would expose full API-level access to every user's financial accounts. Beyond the security concern, the most impactful areas are the unsafe type casts and bare `RuntimeException` throws in the Plaid backend layer, missing schema constraints on `plaid_items.user_id`, and a near-total absence of automated tests across security-critical paths. Groups are ordered: security first, then schema correctness, then backend bugs, then API/exception hygiene, then Angular compliance, then frontend style, then test coverage last.

### Group 1: Security

**Severity**: High  
**Estimated scope**: Large ~3+ hrs  
**Rationale**: All findings in this group are security vulnerabilities (credential exposure, SQL injection risk, authentication bypass risk) that must be resolved before other work proceeds.

#### Tasks

**Task 1.1 — Encrypt `plaid_items.access_token` at rest**
- **File**: `backend/src/main/resources/db/changelog/` (new migration file)
- **What**: Create a new Liquibase changeset that installs the `pgcrypto` extension and alters the `access_token` column handling. Because column type cannot be changed in place for encrypted values, the recommended approach is: (1) add `pgcrypto` via `CREATE EXTENSION IF NOT EXISTS pgcrypto;`, (2) add a new column `access_token_encrypted BYTEA`, (3) backfill it with `pgp_sym_encrypt(access_token, '<app_secret>')`, (4) drop the old `access_token` column, (5) rename `access_token_encrypted` to `access_token`. Update the Spring application layer (`PlaidItemRepository` / `PlaidService`) to encrypt on write and decrypt on read using the same symmetric key sourced from an environment variable (not hardcoded).
- **Why**: `access_token` is a live Plaid API credential. Plain-text storage means a database breach gives full API access to every linked bank account (finding: Section 4 and Section 7, both rated High).
- **Verification**: After migration, query `SELECT access_token FROM plaid_items LIMIT 1;` — the value should be unreadable binary. In the application, confirm Plaid API calls still succeed end-to-end in a local environment.

**Task 1.2 — Validate `orderByClause` against an allowlist before SQL interpolation**
- **File**: `backend/src/main/java/com/killeen/` (the service or mapper layer that passes `orderByClause` to `UserDbMapper.xml`)
- **What**: Locate every call site that populates the `orderByClause` parameter used in the MyBatis mapper `UserDbMapper.xml`. Add an allowlist check — a `Set<String>` of permitted column names (e.g., `{"email", "display_name", "created_at"}`) — before the value reaches the mapper. If the value is not in the allowlist, throw an `IllegalArgumentException` or return results with a default ordering.
- **Why**: `${orderByClause}` in `UserDbMapper.xml` is string interpolation, not a parameterized query. If any user-controlled input ever flows into this parameter, it is a SQL injection vector (finding: Section 4, Medium).
- **Verification**: Confirm the allowlist guard is present at the call site. Attempt to pass `'; DROP TABLE users; --` as the clause — the application should reject it before it reaches the mapper.

### Group 2: Database Schema Integrity

**Severity**: High  
**Estimated scope**: Medium ~1-2 hrs  
**Rationale**: The nullable `user_id` and missing `ON DELETE CASCADE` are data-integrity bugs that will silently produce orphaned rows in production. The missing index will cause full-table scans on every Plaid query. These are grouped as a single schema migration sprint.

#### Tasks

**Task 2.1 — Add `NOT NULL` constraint and `ON DELETE CASCADE` to `plaid_items.user_id`**
- **File**: `backend/src/main/resources/db/changelog/` (new migration file, e.g., `20260329.1-FixPlaidItemsUserIdConstraints.sql`)
- **What**: Create a new Liquibase changeset that performs the following in order: (1) `UPDATE plaid_items SET user_id = <default_or_delete_orphans> WHERE user_id IS NULL;` — either delete orphaned rows (`DELETE FROM plaid_items WHERE user_id IS NULL`) or assign them to a system user; (2) `ALTER TABLE plaid_items ALTER COLUMN user_id SET NOT NULL;`; (3) `ALTER TABLE plaid_items DROP CONSTRAINT IF EXISTS fk_plaid_items_user_id; ALTER TABLE plaid_items ADD CONSTRAINT fk_plaid_items_user_id FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE;`.
- **Why**: `user_id` was added nullable in migration `20260214.2`. Without `NOT NULL`, rows can be inserted with no owner. Without `ON DELETE CASCADE`, deleting a user leaves behind orphaned Plaid items (findings: Section 4 High, Section 7 Medium).
- **Verification**: Insert a `plaid_item` with `user_id = NULL` — expect a constraint violation. Delete a user with linked Plaid items — expect the items to be deleted automatically.

---

**Task 2.2 — Add an index on `plaid_items.user_id`**
- **File**: `backend/src/main/resources/db/changelog/` (can be included in the same migration as Task 2.1)
- **What**: In the same migration changeset as 2.1, add: `CREATE INDEX IF NOT EXISTS idx_plaid_items_user_id ON plaid_items(user_id);`
- **Why**: Every query in `PlaidItemRepository` filters by `user_id`. Without an index, each query performs a full sequential scan of the table (finding: Section 4, Medium).
- **Verification**: Run `EXPLAIN SELECT * FROM plaid_items WHERE user_id = 1;` and confirm an index scan is used.

---

**Task 2.3 — Establish `TEXT` and `TIMESTAMPTZ` as the standard for future migrations**
- **File**: `backend/src/main/resources/db/changelog/` (documentation comment or a new ADR file; existing migrations should NOT be rewritten destructively)
- **What**: Add a comment block at the top of any new migration files going forward: use `TEXT` instead of `character varying(255)` for variable-length string columns with no meaningful max; use `TIMESTAMPTZ` instead of `timestamp without time zone` for all `created_at`/`updated_at` columns. No changes to existing migrations — this is a forward-only convention.
- **Why**: `VARCHAR(255)` imposes an arbitrary cap with no performance benefit in PostgreSQL; `timestamp without time zone` is ambiguous for deployments serving multiple timezones (findings: Section 4, both Medium).
- **Verification**: The next migration file created uses `TEXT` and `TIMESTAMPTZ` for the appropriate column types.

---

**Task 2.4 — Add intent comment to nullable `display_name` column**
- **File**: `backend/src/main/resources/db/changelog/20260214.1-CreateUsersTable.sql`
- **What**: Add a Liquibase `remarks` attribute or an inline `COMMENT ON COLUMN users.display_name IS 'Optional; NULL means the user has not set a display name.';` in a follow-up changeset so the intent is explicit for future maintainers.
- **Why**: Without a comment, it is unclear whether `NULL` is intentional or an oversight (finding: Section 4, Low).
- **Verification**: `\d+ users` in psql shows the column comment.

---

**Task 2.5 — Add Liquibase `rollback` sections to all changesets**
- **File**: All files under `backend/src/main/resources/db/changelog/`
- **What**: For each existing changeset that creates a table, add a `--rollback DROP TABLE <name>;` comment block (or the XML equivalent `<rollback>` tag). For changesets that add columns, add `--rollback ALTER TABLE <name> DROP COLUMN <col>;`. Focus first on the most recently added changesets.
- **Why**: No changesets have rollback instructions, making automated rollback via `liquibase rollback` impossible (finding: Section 7, Low).
- **Verification**: `liquibase rollbackCount 1` executes without error in the local environment.

**Task 3.3 — Eliminate unsafe `@SuppressWarnings("unchecked")` casts in `PlaidController`**
