## Refactor: Move Config Derivation Logic into the Strategy

### Problem

Business logic is currently leaking from `PlaidDataSourceStrategyService` into `PlaidConfigComponent`. The component's `effect()` decides what date range to apply based on the selected metric (e.g. `topMerchantsBySpend` gets a hardcoded 30-day window), and it also decides the `transformConfig.method` based on `visualizationType`. This is strategy-level knowledge living in a UI component.

### Goal

The component should emit only what the user explicitly selects. The strategy should own all config derivation.

### Steps

1. **Add a `resolveConfig()` method to `DataSourceStrategy`** (in `data-source-strategy.interface.ts`)
    * Signature: `resolveConfig(selections: UserSelections): { queryConfig: DataQueryConfig; transformConfig: DataTransformConfig }`
    * `UserSelections` can start as a simple object (e.g. `{ metric, institutionId, visualizationType }`) scoped per data source via generics or a union type

2. **Implement `resolveConfig()` in `PlaidDataSourceStrategyService`**
    * Move the date-range logic (30-day window for `topMerchantsBySpend`, card's current dates otherwise) out of the component and into this method
    * Move the `vizType → method` decision (e.g. `PIE_CHART → accountsByBalance`) out of the component and into this method

3. **Slim down `PlaidConfigComponent`**
    * The component's `effect()` should only emit a `UserSelections` object reflecting the current signal state
    * Remove all `queryConfig`/`transformConfig` construction from the component
    * Update `DataSourceConfigOutput` (or introduce a parallel `UserSelectionsOutput`) so the dashboard card can call `strategy.resolveConfig()` before saving

4. **Update `DashboardCardComponent`**
    * On `saveConfig()`, call `strategy.resolveConfig(draft)` to produce the final `queryConfig` and `transformConfig` before writing to the card
