# Product Plan

## Context

We have a working sandbox Plaid integration: users can link institutions via Plaid Link, and the dashboard displays account balances and transaction data. However, **all data is global and aggregate** — there is no concept of a logged-in user, and there is no way to view data scoped to a specific linked institution. These are the two gaps that need to be closed before the product is useful beyond a single-person demo.

---

## Phase 1: User Management

### Product Goal

A user should be able to create an account, log in, and have their linked institutions and data scoped exclusively to them. No user should ever see another user's connected sources or financial data.

### What This Unlocks

- **Multi-tenancy**: The app can serve more than one person. Each user owns their own set of linked institutions.
- **Session persistence**: A user returns and sees their previously connected sources without re-linking.
- **Foundation for sharing**: Down the road, users could share specific views or invite collaborators — but that requires user identity to exist first.

### User Experience

1. First visit → lands on a sign-up / login screen.
2. After authenticating → redirected to the landing page (existing flow).
3. All Plaid Link connections are saved against the authenticated user.
4. Logging out clears session; logging back in restores their sources and dashboard state.

### Engineering Options

#### Option A: Simple username/password auth (Spring Security + sessions)

- Add a `users` table (id, email, password_hash, created_at).
- Use Spring Security's built-in form login or a custom `/auth/login` + `/auth/register` API.
- Server-side sessions (stored in-memory or in PostgreSQL).
- Frontend stores a session cookie; Angular route guards protect authenticated pages.
- Foreign-key `plaid_items.user_id` ties institutions to the user.

**Pros**: Simplest to build. No external dependencies. Full control.
**Cons**: You own the password storage, reset flow, and security surface. Sessions are server-bound (sticky sessions or shared store needed if you scale horizontally).

#### Option B: JWT-based auth (stateless)

- Same `users` table and registration flow.
- On login, backend issues a signed JWT (access token + optional refresh token).
- Frontend stores the JWT (httpOnly cookie or in-memory) and attaches it to every request via an Angular HTTP interceptor.
- Backend validates the JWT on each request (no server-side session needed).

**Pros**: Stateless — scales horizontally without session replication. Well-suited to SPAs.
**Cons**: Slightly more complex (token refresh, revocation). Still own password management.

#### Option C: Delegated auth via OAuth 2.0 / OpenID Connect (e.g., Google, GitHub)

- Use an identity provider (Google, GitHub, etc.) for authentication.
- Spring Security OAuth2 client handles the redirect flow.
- On first login, create a `users` row from the provider profile (email, name, provider_id).
- Issue a session or JWT after the OAuth callback.

**Pros**: No password management at all. Users get one-click login. Trusted providers handle security.
**Cons**: Depends on external providers. More complex redirect/callback plumbing. Users without a supported provider account can't sign up (unless you offer a fallback).

#### Option D: Third-party auth service (Auth0, Clerk, Firebase Auth, Supabase Auth)

- Offload the entire auth flow (sign-up, login, password reset, MFA) to a managed service.
- Frontend uses their SDK for login UI and token retrieval.
- Backend validates tokens issued by the service.
- Map their user ID to a local `users` row.

**Pros**: Fastest to production-ready auth. Built-in password reset, MFA, social login. Minimal security surface to maintain.
**Cons**: External dependency and cost. Less control over the experience. Vendor lock-in risk.

### Recommendation

For a project at this stage, **Option A or B** keeps everything self-contained and avoids external dependencies while the product is still being shaped. Option B (JWT) is a slight favorite since the app is an SPA and stateless auth fits the architecture naturally. If speed-to-market matters more than control, **Option D** (managed service) is worth considering — but it adds a dependency before the product direction is fully validated.

## Phase 2: Per-Institution Data Visualization

### Product Goal

A user should be able to see their financial data broken down by linked institution rather than only in a single aggregate view. They should be able to toggle between an "all accounts" aggregate view and a per-institution drill-down.

### What This Unlocks

- **Clarity**: When a user links a checking account at Bank A and a credit card at Bank B, they can see each institution's transactions and balances independently.
- **Debugging / trust**: Users can verify that a specific institution's data looks correct without noise from others.
- **Richer analysis**: Compare spending patterns across institutions. Identify which accounts drive which behaviors.

### User Experience

1. Dashboard defaults to the **aggregate** view (current behavior — all institutions combined).
2. A filter or selector (dropdown, tab bar, or sidebar) lets the user choose a specific linked institution.
3. When an institution is selected, all charts and data on the dashboard scope to that institution's accounts and transactions only.
4. An "All Institutions" option returns to the aggregate view.
5. (Future) Side-by-side comparison of two institutions.

### Engineering Options

#### Option A: Frontend-only filtering

- The backend already attaches `institutionId` to each `DataPoint` metadata.
- Backend continues to return all data in one response.
- Frontend filters the data by institution before passing it to chart components and transformers.
- Add an institution selector component that reads from the list of connected items.

**Pros**: Zero backend changes. Fast to implement. All logic lives in the transformers and dashboard components.
**Cons**: Sends all data over the wire even when viewing a single institution. Doesn't scale if a user has many institutions with large transaction histories.

#### Option B: Backend query-parameter filtering

- Add an optional `institutionId` (or `itemId`) query parameter to `/plaid/transactions` and `/plaid/account-balances`.
- When provided, the backend only fetches data for that specific Plaid item instead of iterating over all items.
- Frontend sends the filter as part of the API call based on the user's selection.

**Pros**: Reduces payload size and API call overhead. Keeps the backend's data-fetching efficient. Clean separation of concerns.
**Cons**: Requires backend changes (though minor — conditional filtering in `PlaidService.fetchData()`).

#### Option C: Cached / stored data with backend grouping

- Instead of fetching from Plaid on every request, periodically sync and store transactions and balances locally (in PostgreSQL).
- Backend can then group, filter, and paginate server-side with SQL.
- API returns data pre-grouped by institution, date range, account, etc.

**Pros**: Fastest queries. Enables complex aggregations, historical analysis, and search. Reduces Plaid API call volume.
**Cons**: Significant backend work (sync jobs, data staleness management, storage growth). Overkill at this stage unless data volume is a concern.

### Recommendation

Start with **Option A** (frontend filtering) since the metadata is already there and it requires the least effort. Move to **Option B** once the feature is validated and you want to optimize network/performance. **Option C** is the long-term direction if the product grows to support heavy historical analysis or many institutions per user — but it's premature until then.

---

## Sequencing Summary

| Phase | Goal | Depends On | Rough Scope |
|-------|------|------------|-------------|
| **1 — User Management** | Scoped, authenticated experience | Nothing (foundational) | `users` table, auth flow, route guards, FK on `plaid_items` |
| **2 — Per-Institution Views** | Granular data exploration | Phase 1 (data is per-user) | Institution selector, filtered transformers/charts |

Phase 1 must come first: without user identity, per-institution views are meaningless because all institutions are shared globally. Once users own their data, scoping that data by institution is a natural next layer.
