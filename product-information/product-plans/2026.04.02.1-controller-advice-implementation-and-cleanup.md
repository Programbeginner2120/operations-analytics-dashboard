# Controller Advice Implementation & Cleanup
**Date:** 2026-04-02

## Objective

Replace inline `try-catch` blocks and bare `RuntimeException` throws with a centralized `@ControllerAdvice` (`GlobalExceptionHandler`). Controllers should delegate to services and return results — all exception-to-HTTP-status mapping lives in one place.

---

## Background

**Current problems:**
- `AuthController.register()` and `login()` catch `IllegalArgumentException` inline and manually return status codes.
- `UserService` and `EmailTokenService` throw the generic `IllegalArgumentException` for multiple distinct failure modes — a global handler cannot distinguish them without semantic exception types.
- `PlaidService` throws bare `RuntimeException` throughout.
- `AuthController` return types are `ResponseEntity<?>` (wildcard) because error bodies differ from success bodies — this is a symptom of inline error handling.
- No consistent API error response shape exists.

---

## Step 1 — Create Domain-Specific Exception Classes

Create the following exceptions. Each is a `RuntimeException` subclass with a `(String message)` constructor.

**`components/user/exception/`**
- `UserAlreadyExistsException` — duplicate email on registration
- `InvalidCredentialsException` — wrong email or password on login
- `UserNotFoundException` — user lookup by ID fails

**`components/email/exception/`** *(package already exists)*
- `InvalidTokenException` — token unknown, expired, already used, or wrong type (replaces four `IllegalArgumentException` throws in `EmailTokenService`)

**`components/plaid/exception/`**
- `PlaidOperationException` — wraps Plaid API failures and IO errors (replaces bare `RuntimeException` in `PlaidService`)
- `PlaidItemNotFoundException` — item not found on delete (enables a 404 response distinct from 500)

---

## Step 2 — Create `ErrorResponse` DTO

Create `exception/ErrorResponse.java` (a simple record or class) at the top-level `exception` package:

```java
package com.killeen.dashboard.exception;

public record ErrorResponse(int status, String message) {}
```

This gives every error response a consistent JSON shape:
```json
{ "status": 409, "message": "An account with this email already exists" }
```

---

## Step 3 — Create `GlobalExceptionHandler`

Create `exception/GlobalExceptionHandler.java` annotated with `@RestControllerAdvice`.

Exception-to-status mapping:

| Exception | HTTP Status | Notes |
|---|---|---|
| `UserAlreadyExistsException` | 409 Conflict | |
| `InvalidCredentialsException` | 401 Unauthorized | Use generic message for security |
| `UserNotFoundException` | 404 Not Found | |
| `EmailNotVerifiedException` | 403 Forbidden | |
| `InvalidTokenException` | 400 Bad Request | |
| `PlaidItemNotFoundException` | 404 Not Found | |
| `PlaidOperationException` | 502 Bad Gateway | |
| `DataSourceException` | 502 Bad Gateway | |
| `MethodArgumentNotValidException` | 400 Bad Request | Collect all field errors into message |
| `Exception` (fallback) | 500 Internal Server Error | Log at ERROR level; return generic message (do not leak internal details) |

Each handler method logs at an appropriate level and returns `ResponseEntity<ErrorResponse>`.

## Step 4 — Update Services to Throw Semantic Exceptions

### `UserService`
| Current throw | Replace with |
|---|---|
| `throw new IllegalArgumentException(env.getProperty("user.email.already.exists"))` in `register()` | `throw new UserAlreadyExistsException(...)` |
| `throw new IllegalArgumentException(env.getProperty("user.credentials.invalid"))` in `authenticate()` | `throw new InvalidCredentialsException(...)` |
| `throw new IllegalArgumentException(env.getProperty("user.not.found"))` in `getUserById()` | `throw new UserNotFoundException(...)` |
| `throw new IllegalArgumentException(env.getProperty("email.no.account.found.with.email"))` in `resendVerification()` | `throw new UserNotFoundException(...)` |

### `EmailTokenService`
| Current throw | Replace with |
|---|---|
| All four `throw new IllegalArgumentException(...)` in `validateAndConsume()` | `throw new InvalidTokenException(...)` |

### `PlaidService`
| Current throw | Replace with |
|---|---|
| `throw new RuntimeException(...)` in `createLinkToken()` | `throw new PlaidOperationException(...)` |
| `throw new RuntimeException(...)` in `exchangePublicToken()` | `throw new PlaidOperationException(...)` |
| `throw new RuntimeException(String.format(...), itemId)` in `deleteItem()` | `throw new PlaidItemNotFoundException(...)` |
| `throw new RuntimeException(...)` in `fetchDataForItem()` | `throw new PlaidOperationException(...)` |
| `throw new RuntimeException(...)` in `fetchDataWithSandboxToken()` | `throw new PlaidOperationException(...)` |

## Step 5 — Clean Up `AuthController`

### `register()`
- Remove the `try-catch` block.
- Tighten return type from `ResponseEntity<?>` to `ResponseEntity<UserResponse>`.

### `login()`
- Remove the `try-catch` block.
- Tighten return type from `ResponseEntity<?>` to `ResponseEntity<LoginResponse>`.

### `getCurrentUser()`
- Remove the manual `authentication == null || !authentication.isAuthenticated()` guard. The `JwtAuthenticationFilter` already rejects unauthenticated requests before they reach this method; the check is dead code on secured endpoints.
- The `(Long) authentication.getPrincipal()` cast remains, consistent with the filter's behaviour.

Also remove the `@Slf4j` annotation and `log` field from `AuthController` — logging belongs in the advice and the service, not the controller.

---

## Step 6 — Clean Up `PlaidController`

### `getAuthenticatedUserId()`
- The helper throws `ResponseStatusException` directly. This is fine as-is since Spring handles `ResponseStatusException` natively, but for consistency it can be left alone — the advice does not need to handle it.
- No other changes required; `PlaidController` already has no try-catch blocks.
